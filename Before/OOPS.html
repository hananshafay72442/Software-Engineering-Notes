Basic OOP Concepts

Fundamentals

Classes · Objects · The Four Pillars · Pointers · Access Specifiers

Pillar 1
Encapsulation
  • Bundles data + methods into a single class unit
  • Restricts direct access to internal state using access specifiers
  • Data is hidden (private); only exposed via public methods (getters/setters)
  • Benefit: protects integrity of object data; changes to internals don't break outside code
Pillar 2
Abstraction
  • Exposes what an object does, hides how it does it
  • Shows only essential features to the outside world
  • Achieved via abstract classes and interfaces
  • Reduces complexity — user interacts with a simplified model
Pillar 3
Inheritance
  • A child (derived) class acquires properties and behavior of a parent (base) class
  • Establishes an "is-a" relationship — e.g. Dog is-a Animal
  • Promotes code reuse — no need to rewrite common logic
  • Child can extend (add new) or override (replace) parent behavior
Pillar 4
Polymorphism
  • One interface, many forms — same function name, different behavior
  • Compile-time: function/operator overloading — resolved by compiler
  • Runtime: function overriding via virtual functions — resolved at runtime via vtable
  • Allows treating objects of different classes uniformly through a common interface

private:
  double balance; // hidden
public:
  BankAccount(double b) : balance(b) {}

// balance is only changed via methods
Inheritance — C++
class Animal {
public:
  void breathe() {
    cout << "Breathing...";
  }
};

class Dog : public Animal {
public:
  void bark() {
    cout << "Woof!";
  }
};
// Dog inherits breathe() from Animal
Polymorphism (Runtime) — C++
class Shape {
public:
  virtual void draw() {
    cout << "Shape";
  }
};
class Circle : public Shape {
public:
  void draw() override {
    cout << "Circle";
  }
};
Shape* s = new Circle();
s->draw(); // prints "Circle" ← runtime!
Abstraction — Abstract Class C++
class Vehicle {  // Abstract
public:
  virtual void start() = 0; // pure virtual
  void stop() {
    cout << "Stopped";
  }
};
class Car : public Vehicle {
public:
  void start() override {
    cout << "Car started";
  }
};
// Vehicle* v = new Vehicle(); ← ERROR!
  • A Class is a blueprint / template — it defines data members and member functions but holds no memory itself
  • An Object is a runtime instance of a class, allocated in memory with its own copy of data members
  • Example: class Car { }; is the blueprint; Car myCar; creates an object
  • Multiple objects can be created from one class — each has its own state
Access Specifiers
  • private — accessible only within the class. Default in class
  • protected — accessible within the class and derived classes
  • public — accessible from anywhere
  • In a struct, members are public by default
  • Core mechanism of Encapsulation
Pointers — Basics
Address-of & Dereference
  • &x — address-of operator: returns the memory address of x
  • int* p = &x; — pointer p stores the address of x
  • *p — dereference: follows the address in p and returns the value
  • Changing *p changes the original variable x
Pointers — Member Access
Dot vs Arrow Operator
  • obj.member — dot operator used on a direct object instance
  • ptr->member — arrow operator used when accessing through a pointer
  • ptr->speed is shorthand for (*ptr).speed
  • :: (scope resolution) is used to access class members outside the class definition, or to access static members

Pointers — Memory Visualization
How & and * work in memory
Pointers — C++
int x = 42;
int* p = &x;     // p holds address of x

cout << x;      // 42  (direct)
cout << &x;     // 0x1A4 (address)
cout << p;      // 0x1A4 (same address)
cout << *p;     // 42   (dereference)

*p = 99;        // changes x!
cout << x;      // 99

// Arrow operator with objects:
Car* car = new Car();
car->start();   // same as (*car).start()

C++ & OOP In Depth

C++

Constructors · Destructors · Friend Functions · Inheritance · Polymorphism · Templates

Constructors
Types of Constructors
  • Default — no parameters; called when object is created with no args
  • Parameterized — accepts arguments to initialize members with specific values
  • Copy constructor — signature: MyClass(const MyClass& other); invoked when initializing from another object, passing by value, or returning by value
  • Same name as class; no return type (not even void)
  • Constructors can be overloaded
Destructors
Destructor
  • Prefixed with ~: e.g. ~MyClass()
  • Called automatically when an object goes out of scope or is deleted
  • Purpose: release resources — free heap memory, close file handles, release locks
  • A class has only one destructor — cannot be overloaded
  • Always declare virtual destructor in base classes to ensure correct cleanup for derived objects
Friend Functions
Friend Function
  • Declared inside the class with the friend keyword
  • Not a member of the class — defined outside
  • Has access to private and protected members of the class
  • Breaks encapsulation intentionally — commonly used for operator overloading
  • Friendship is not inherited — not transitive
All Constructor Types — C++
class Box {
  int w, h;
public:
  // 1. Default constructor
  Box() : w(0), h(0) {}

  // 2. Parameterized constructor
  Box(int w, int h) : w(w), h(h) {}

  // 3. Copy constructor
  Box(const Box& b) : w(b.w), h(b.h) {}

  // Destructor
  ~Box() { /* cleanup */ }
};

Box b1;          // default
Box b2(5, 10);  // parameterized
Box b3 = b2;    // copy constructor
Friend Function — C++
class Counter {
  int count = 0;
public:
  // friend declaration inside class
  friend void showCount(Counter& c);
};

// defined OUTSIDE the class
void showCount(Counter& c) {
  // accesses private 'count'
  cout << "Count: " << c.count;
}

Counter c;
showCount(c); // Count: 0
// not a member — called without object!
  • One child inherits from one parent
  • class B : public A { };
  • Simplest form; clean hierarchy
Multiple Inheritance
  • One child inherits from two or more parents
  • class C : public A, public B { };
  • Can cause Diamond Problem — solved with virtual base classes
  • Java does not support this for classes
Multilevel & Hierarchical
  • Multilevel: chain — A→B→C (grandparent/parent/child)
  • Hierarchical: one base, many children — e.g. Animal → Dog, Cat, Bird
  • Hybrid: combination of multiple types

Templates — C++
// Generic function — works with any type
template<typename T>
T getMax(T a, T b) {
  return (a > b) ? a : b;
}

cout << getMax(3, 7);      // 7   (int)
cout << getMax(3.5, 2.1); // 3.5 (double)

template<typename T>
class Stack {
  T data[100];
  int top = -1;
public:
  void push(T val) { data[++top] = val; }
  T pop() { return data[top--]; }
};
Virtual Functions & vtable — C++
class Base {
public:
  virtual void speak() {
    cout << "Base speaks";
  }
  virtual ~Base() {} // virtual destructor!
};

class Derived : public Base {
public:
  void speak() override {
    cout << "Derived speaks";
  }
};

Base* ptr = new Derived();
ptr->speak(); // "Derived speaks"
// vtable lookup at RUNTIME
Function & Operator Overloading
  • Function overloading: same name, different parameter types/count — resolved by compiler
  • Operator overloading: redefine built-in operators for user-defined types using operator+, operator==, etc.
  • Example: Point p3 = p1 + p2; calls operator+
  • Also called static polymorphism
Runtime Polymorphism
Virtual Functions & Abstract Classes
  • Declare base function as virtual → enables dynamic dispatch via vtable
  • Correct overridden version runs based on actual object type at runtime
  • Pure virtual: virtual void func() = 0; — must be overridden in derived classes
  • A class with at least one pure virtual function is an abstract class — cannot be instantiated
Templates
Function & Class Templates
  • Write generic code that works with any data type
  • Syntax: template<typename T>
  • Compiler generates a version for each type used — called template instantiation
  • Avoids code duplication for int, float, string, etc.
  • Also applies to class templates (e.g. std::vector<T>)

Java Fundamentals

Java

JVM · JIT · JDK · JRE · Data Types · OOP in Java · Memory Model

JDK / JRE / JVM — What's what
  • JDK (Java Development Kit) — full developer toolkit: includes javac (compiler), javadoc, jar, debugger + JRE. Needed to develop
  • JRE (Java Runtime Environment) — libraries + JVM. Needed to run Java programs. No compiler
  • JVM (Java Virtual Machine) — executes bytecode. Platform-specific; bytecode is not
  • Compilation flow: .java → javac → .class (bytecode) → JVM → native machine code
Write Once, Run Anywhere
  • javac compiles source to platform-independent bytecode (.class)
  • The bytecode is identical on all platforms — the JVM is what's platform-specific
  • Each OS has its own JVM implementation that translates bytecode to native CPU instructions
  • JIT Compiler (Just-In-Time): part of the JVM's Execution Engine — profiles running code, finds "hot spots", compiles them to native machine code at runtime for speed
JVM Components
Inside the JVM
  • Class Loader — loads .class files from disk into memory
  • Execution Engine — runs bytecode; contains Interpreter + JIT compiler
  • Garbage Collector — automatically frees heap memory for unreferenced objects; no manual delete needed (unlike C++)
  • Runtime Data Areas — memory regions: Heap, Stack, Method Area, PC Register
JVM Memory Model
Heap, Stack & Method Area
  • Heap — stores all objects and class instances; managed by GC
  • Stack — stores method call frames and local variables; one per thread
  • Method Area — stores class metadata, static fields, and constants
  • PC Register — program counter; tracks current instruction per thread
Data Types
Java Primitive Types
Java has 8 primitive types:

byte short int long float double char boolean

  • char — 16-bit Unicode character
  • String is a class, not a primitive
    • Java OOP
    Inheritance & Keywords
    • extends — create a child class: class Dog extends Animal
    • Java supports single class inheritance only — no multiple inheritance for classes
    • super — calls parent constructor or method
    • implements — implement an interface
    • A class can implement multiple interfaces
    Java Interfaces
    Interface vs Abstract Class
    • Interface: a contract of method signatures a class must implement; enables multiple type inheritance
    • Pre-Java 8: no method bodies; post-Java 8: can have default and static methods
    • Abstract class: cannot be instantiated; may have both abstract and concrete methods
    • A class can implement many interfaces but extend only one abstract class
    Access Modifiers
    Visibility in Java
    Visibility from most to least restrictive:

    • private — class only
    • package-private (no modifier) — same package
    • protected — package + subclasses
    • public — everywhere

    Question 1 · 12 Marks
    Part A · CLO1 3 Marks

    Encapsulation & Abstraction

    Explain the concept of Encapsulation and Abstraction with suitable examples.

    ✦ Encapsulation

    Encapsulation means wrapping data (variables) and functions together inside a class, and hiding the data from outside access. We use private to protect the data and public functions to access or change it.

    Real-life example: A bank account hides the balance. You cannot directly change it — you must use deposit() or withdraw().

    class BankAccount { private: int balance; // hidden from outside public: BankAccount() { balance = 0; } void deposit(int amount) { balance += amount; } int getBalance() { return balance; } }; int main() { BankAccount acc; acc.deposit(500); cout << acc.getBalance(); // 500 }

    ✦ Abstraction

    Abstraction means showing only the necessary details to the user and hiding the internal implementation. The user knows what a function does, not how it does it.

    Real-life example: When you drive a car, you use the steering wheel and pedals. You do not need to know how the engine works internally.

    class Car { private: void startEngine() { // hidden internal detail cout << "Engine started"; } public: void drive() { // user only sees this startEngine(); cout << "Car is moving"; } }; int main() { Car myCar; myCar.drive(); // user does not know about startEngine() }
    💡 Key Difference: Encapsulation is about protecting data. Abstraction is about hiding complexity.
    Part B · CLO1 3 Marks

    Function Overloading vs Function Overriding

    Differentiate between Function Overloading and Function Overriding with examples.

    Overloading

    Same function name, different parameters, in the same class. Decided at compile time.

    Overriding

    Same function name, same parameters, in parent & child class. Decided at run time.

    Example — Overloading

    class Calculator { public: int add(int a, int b) { return a + b; } double add(double a, double b) { // same name, different params return a + b; } }; int main() { Calculator c; cout << c.add(2, 3); // calls first add → 5 cout << c.add(2.5, 3.1); // calls second add → 5.6 }

    Example — Overriding

    class Animal { public: virtual void sound() { cout << "Some animal sound"; } }; class Dog : public Animal { public: void sound() { // overrides parent function cout << "Woof!"; } }; int main() { Animal* a = new Dog(); a->sound(); // prints: Woof! (Dog's version) }
    Part C · CLO2 3 Marks

    Library Management System

    Identify three objects (classes), their attributes and functions, and describe the relationship between two of them.

    📗 Book

    Attribute: title (string)

    Function: getTitle()

    👤 Member

    Attribute: name (string)

    Function: borrowBook()

    📚 Library

    Attribute: libraryName (string)

    Function: addBook()

    Relationship

    A Library has many Books — this is a Has-A relationship (Composition/Aggregation). The Library class contains Book objects. If the library is closed, the books still exist, so it is Aggregation.

    class Book { public: string title; string getTitle() { return title; } }; class Library { public: string libraryName; Book books[100]; // Library HAS Books (aggregation) int count = 0; void addBook(Book b) { books[count++] = b; } };
    Part D · CLO2 3 Marks

    Code Output Prediction

    Predict the output and explain the reason behind the behavior.

    Given Code

    class Student { public: string name; int marks; }; void modify(Student s) { // pass by VALUE s.marks = 100; } int main() { Student s1 = {"Ali", 70}; modify(s1); cout << s1.marks; }

    ✅ Output

    70

    📖 Explanation

    The function modify(Student s) receives the object by value. This means C++ creates a copy of s1 and sends that copy to the function.

    When we write s.marks = 100 inside the function, we are changing the copy, not the original s1. So s1.marks remains 70.

    To actually change s1.marks, you would need to pass by reference:

    void modify(Student& s) { // pass by REFERENCE s.marks = 100; // now changes the original } // Output would then be: 100

    Question 2 · 08 Marks

    📗 University Library Book Issuing System

    Estimated time: 40 minutes  ·  CLO3  ·  8 × 1 = 08 Marks  ·  Concepts: Classes, Inheritance, Polymorphism, Encapsulation

    Part (a) · CLO3 Book Class

    Designing the Book Class

    We create a class Book with private data members (bookID, title, author) and public member functions (setData(), display()).

    #include <iostream> #include <string> using namespace std; class Book { private: int bookID; // private — cannot be accessed from outside string title; string author; public: // setData() takes input and stores it in private members void setData(int id, string t, string a) { bookID = id; title = t; author = a; } // display() shows the book information void display() { cout << "Book ID : " << bookID << endl; cout << "Title : " << title << endl; cout << "Author : " << author << endl; } };
    Part (b) · CLO3 Inheritance & Polymorphism

    Creating IssuedBook — Inheritance & Polymorphism

    🔗 Inheritance

    IssuedBook inherits from Book using : public Book. This means it gets all public functions of Book automatically — no need to re-write them.

    🔄 Polymorphism

    The display() function is overridden in IssuedBook to also print the issue date — same function name, new behaviour in child class.

    class IssuedBook : public Book { // IssuedBook inherits from Book private: string issueDate; // extra data member only in IssuedBook public: // setIssueDate() stores the issue date void setIssueDate(string date) { issueDate = date; } // display() is OVERRIDDEN to also show issue date void display() { Book::display(); // call parent's display first cout << "Issue Date: " << issueDate << endl; } };
    Part (c) · CLO3 Encapsulation

    Encapsulation — Keeping Data Safe

    In both classes, all data members are declared private. This means no code outside the class can directly read or change them. They can only be accessed through the public member functions we provide.

    ❌ Without Encapsulation

    Anyone can write b1.bookID = -99; and corrupt the data. No control, no safety.

    ✅ With Encapsulation

    Data is private. Only setData() can change it — full control over what goes in.

    Rule: Data members → private. Functions that access them → public. This is the foundation of Encapsulation.

    Part (d) · CLO3 main() Demonstration

    Complete Program — main() Function

    Below is the full working program combining all parts. It creates one object of Book and one of IssuedBook, inputs their data, and displays them.

    #include <iostream> #include <string> using namespace std; // ── Base Class ────────────────────────────────────────── class Book { private: int bookID; string title; string author; public: void setData(int id, string t, string a) { bookID = id; title = t; author = a; } void display() { cout << "Book ID : " << bookID << endl; cout << "Title : " << title << endl; cout << "Author : " << author << endl; } }; // ── Derived Class ─────────────────────────────────────── class IssuedBook : public Book { private: string issueDate; public: void setIssueDate(string date) { issueDate = date; } void display() { // overrides Book::display() Book::display(); // print base class info first cout << "Issue Date: " << issueDate << endl; } }; // ── main() ────────────────────────────────────────────── int main() { // Object of Book (base class) Book b1; b1.setData(101, "Introduction to C++", "Bjarne Stroustrup"); cout << "--- Regular Book ---" << endl; b1.display(); // Object of IssuedBook (derived class) IssuedBook ib1; ib1.setData(202, "Object Oriented Programming", "Robert Lafore"); ib1.setIssueDate("31-March-2026"); cout << "\n--- Issued Book ---" << endl; ib1.display(); // calls IssuedBook's display() — polymorphism return 0; }

    ✅ Expected Output

    --- Regular Book ---
    Book ID : 101
    Title : Introduction to C++
    Author : Bjarne Stroustrup

    --- Issued Book ---
    Book ID : 202
    Title : Object Oriented Programming
    Author : Robert Lafore
    Issue Date: 31-March-2026

    🔗 Relationship Summary

    Book is the parent (base) class. IssuedBook is the child (derived) class. This is an IS-A relationship — every IssuedBook is a Book, plus it has extra issue details. The child inherits setData() from the parent and overrides display() to add its own information.
